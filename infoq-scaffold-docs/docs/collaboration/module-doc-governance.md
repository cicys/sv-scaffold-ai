# 模块文档分层治理规范

本文档用于约束仓库内各工作区的模块文档如何分层、如何聚合、何时同步，目标不是“把所有细节写成百科全书”，而是让人和 AI 都能按稳定路径找到最接近代码的真值，减少误判和文档漂移。

## 1. 适用范围

适用于存在明显层级拆分的工作区，例如：

- `infoq-scaffold-backend`
- `infoq-scaffold-frontend-react`
- `infoq-scaffold-frontend-vue`
- `infoq-scaffold-frontend-weapp-react`
- `infoq-scaffold-frontend-weapp-vue`

如果某个工作区内部还分为父模块与叶子模块，这份规范优先采用“叶子真值 -> 父层聚合 -> 工作区总览 -> AGENTS 路由”的结构。

## 2. 核心结论

更利于 AI 理解项目的方式，不是只维护一份顶层总览，也不是把所有说明堆进一个 README，而是分层治理：

1. 叶子模块文档负责写真值，直接贴近代码。
2. 父模块文档负责聚合子模块职责、依赖方向和调用关系。
3. 工作区顶层文档负责总览、阅读路径和关键架构 / 数据流摘要。
4. 工作区 `AGENTS.md` 负责把这些文档注册成稳定的检索入口，而不是重复承载文档正文。

这套结构的关键不是“文档更多”，而是“真值更近、聚合更克制、路由更明确”。

## 3. 推荐分层

| 层级 | 推荐位置 | 主要职责 |
| --- | --- | --- |
| 叶子模块文档 | 叶子模块目录下的 `README.md` 或 `doc/*.md` | 描述当前实现、入口、依赖、配置、关键数据流、日志/监控切入点 |
| 父模块聚合文档 | 父模块目录下的 `README.md` 或 `doc/*.md` | 梳理子模块边界、依赖方向、调用关系、公共约束 |
| 工作区顶层文档 | 工作区 `README.md`、`doc/architecture.md`、`doc/data-flow.md` 等 | 给出总览、阅读顺序、跨模块架构图和关键业务流 |
| AI 路由文档 | 工作区 `AGENTS.md` | 指明优先阅读的文档与同步门禁 |

以 backend 为例，理想结构是：

1. 先在 `infoq-core`、`infoq-modules`、`infoq-plugin` 的叶子模块下补真值文档。
2. 再在 `infoq-core`、`infoq-modules`、`infoq-plugin` 这一层写聚合文档。
3. 最后由 `infoq-scaffold-backend/README.md`、`doc/architecture.md`、`doc/data-flow.md` 汇总。
4. `infoq-scaffold-backend/AGENTS.md` 只负责告诉 AI 先读什么、变更后要同步哪些层。

前端工作区可以复用同一原则，只是叶子粒度不一定是 Maven 模块，也可以是 feature、package、page 或 domain module。

## 4. 每一层应该写什么

### 4.1 叶子模块文档

叶子模块文档是最重要的真值层，建议至少覆盖：

- 模块职责：这个模块解决什么问题，不解决什么问题。
- 关键入口：启动类、控制器、路由入口、导出入口、自动配置入口、注册点。
- 核心类/文件：当前实现真正关键的类、配置、SQL、页面或适配器。
- 上游依赖：谁会调用它、装配它、引用它。
- 下游依赖：它依赖哪些核心能力、外部服务或共享模块。
- 关键配置：影响行为的开关、环境变量、profile、注册参数。
- 关键数据流：输入从哪来，经过什么层，输出到哪去。
- 扩展点：SPI、hook、strategy、plugin、可替换实现。
- 日志/监控切入点：现有日志类、指标来源、监控接口、排障入口。
- 已知边界：当前仓库能证实的范围，以及不能臆测的部分。

### 4.2 父模块聚合文档

父模块文档不应重写子模块细节，而应聚焦“关系”：

- 子模块清单与各自职责摘要。
- 子模块之间的依赖方向。
- 典型调用链和装配关系。
- 哪些能力属于公共能力，哪些属于业务能力。
- 哪些关系是运行时装配，哪些只是编译期依赖。
- 哪些链路已被代码证实，哪些只能保守写成“当前实现说明”。

### 4.3 工作区顶层文档

工作区顶层文档负责“先让人看懂，再决定往哪里钻”：

- 工作区整体定位与模块分层。
- 推荐阅读路径。
- 总体架构与关键数据流。
- 跨父模块的共性约束，例如安全、配置、插件、部署、监控。
- AI 或新人最容易误判的边界提醒。

### 4.4 `AGENTS.md`

`AGENTS.md` 不代替正文文档，只负责：

- 注册文档入口。
- 规定阅读顺序。
- 规定什么变化必须同步哪一层文档。
- 约束 AI 在信息不足时回退为“当前实现说明”。

## 5. 聚合文档必须理顺调用关系

聚合文档不能只列模块名单，否则 AI 仍然会误判结构。至少要把以下关系理顺：

- 依赖方向：谁依赖谁，是否允许反向依赖。
- 调用入口：请求、消息、任务、自动配置或前端事件从哪里进入。
- 主链路：例如 `controller -> service -> mapper`、`page -> store -> request -> api`、`plugin auto-config -> shared capability -> business module`。
- 横切能力接入方式：安全、日志、缓存、定时任务、字典、审计、监控如何挂入主流程。
- 运行时与编译期区别：有些模块只是被引用，有些模块会在运行期实际装配生效。

如果关系未被代码证实，不要为了完整而脑补，直接写“当前仓库仅能确认到这里”。

## 6. 文档同步门禁

不要把规则写成“任何代码变动都更新所有父文档”，这会快速失控。更合理的同步门禁是：

### 6.1 必须同步文档的变更

- 模块职责变化。
- 对外接口、入口或导出点变化。
- 子模块依赖方向变化。
- 关键调用链或关键数据流变化。
- 关键配置项、开关、profile、环境变量变化。
- 自动配置、装配关系、扩展点变化。
- SQL / 持久化语义变化。
- 现有日志/监控/排障入口变化。

### 6.2 通常不需要层层同步的变更

- 纯命名微调且不影响语义。
- 纯内部重构且职责和边界不变。
- 注释、格式化、小范围测试补充。
- 不改变接口和依赖关系的局部实现细节。

### 6.3 推荐同步顺序

1. 先更新受影响的叶子模块文档。
2. 再更新该叶子模块的直接父模块聚合文档。
3. 若变化已影响工作区总览，再更新工作区顶层文档。
4. 若文档入口、阅读顺序或同步门禁发生变化，再更新该工作区 `AGENTS.md`。

交付前至少做一次“文档漂移检查”：确认上层摘要仍与下层真值一致。

## 7. 冲突裁决规则

多层文档并存时，必须明确真值优先级：

1. 源码与真实配置是最终真值。
2. 离代码最近的叶子模块文档优先。
3. 直接父模块聚合文档次之。
4. 工作区顶层总览文档再次之。
5. `AGENTS.md` 只负责路由和门禁，不作为实现细节真值。

如果多层文档与代码不一致，先以代码为准，并把描述回退为更保守的“当前实现说明”。

## 8. 反模式

以下做法会让 AI 更容易跑偏：

- 每次小改动都强制改完整棵文档树。
- 父模块文档复制粘贴子模块文档正文。
- 文档写满未来规划、猜测链路、未证实依赖。
- 小模块也写成长篇大论，导致维护成本高于信息价值。
- 把调用关系藏在脑子里，只在代码里散落，不在聚合层显式说明。

## 9. 建议模板

叶子模块文档建议使用以下结构：

```md
# 模块名

## 1. 模块职责
## 2. 关键入口
## 3. 核心类 / 文件
## 4. 上游依赖
## 5. 下游依赖
## 6. 关键配置
## 7. 关键数据流
## 8. 扩展点
## 9. 日志 / 监控切入点
## 10. 已知边界
```

父模块聚合文档建议使用以下结构：

```md
# 父模块名

## 1. 模块定位
## 2. 子模块清单
## 3. 子模块职责摘要
## 4. 依赖方向
## 5. 典型调用链
## 6. 公共约束
## 7. 已知边界
## 8. 下钻阅读路径
```

## 10. 落地建议

针对后续工作区治理，建议按下面顺序推进：

1. 先补最容易被误判的叶子模块文档。
2. 再补直接父模块的聚合文档，把调用关系和依赖方向理顺。
3. 最后更新工作区总览文档和 `AGENTS.md` 文档路由。
4. 把“是否需要同步文档”纳入每次交付前的检查项。

如果某条链路暂时梳理不清，不要硬凑完整说明，直接写成保守版本的“当前实现说明”。对 AI 来说，边界清楚的保守描述，优于貌似完整但不真实的泛化总结。
